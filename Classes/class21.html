Domain Adaptation

How do you teach an LLM new domain knowledge — like legal contracts, medical literature, or a new language — without providing any question–answer pairs? Continued Pre-Training (CPT) is the answer: fine-tune on raw text so the model internalises domain vocabulary, facts, and writing patterns before any instruction tuning happens.

02 — Background

What is Non-Instruction Fine-Tuning?

Base models like Llama 3, Mistral, and Qwen are pre-trained on trillions of tokens scraped from the internet. They are extraordinarily general — but that generality has a cost. If your use-case lives in a specialised domain (legal, medical, finance, a low-resource language, a niche codebase), the base model may:

📚

Lack Domain Vocabulary

Rare domain terms may not appear in pre-training data, causing hallucination or misuse of jargon.

🌐

Underrepresent Languages

English dominates training data. Low-resource languages (Swahili, Korean, Tamil) receive far less exposure.

🧬

Miss Recent Knowledge

Training data has a cutoff date. Events, papers, or regulations published after that date are unknown to the model.

✍️

Mismatch in Style

Generating in a specific house style (legal boilerplate, medical notes, formal Japanese) requires exposure to many examples of that style.

Continued Pre-Training (CPT) — also called non-instruction fine-tuning — solves all of the above by running the same next-token prediction objective on your domain corpus. No labels, no question–answer pairs, just raw text.

💡

Think of it this way: If pre-training made the model a well-read generalist, CPT sends that generalist to medical school for a semester. They read thousands of papers, learn the terminology, and absorb the reasoning patterns — before ever being asked to treat a patient (which is SFT).

Real-World Use Cases

CPT is the industry-standard first step when building domain-specific LLMs. Here are concrete examples:

⚖️ Legal Train on court judgements, legislation, and legal commentary to build a contract-analysis model.
🏥 Medical Feed PubMed abstracts, clinical notes, and pharmaceutical documents before SFT for a medical Q&A assistant.
💹 Finance BloombergGPT is a famous example — GPT trained heavily on financial news, filings, and reports.
🌐 Language Teach a largely-English model Korean, Tamil, or Swahili by training on a large corpus of that language.
💻 Code Train on a private codebase so the model understands company-internal APIs and naming conventions.
📖 Style Train on a brand's editorial archive to adopt a specific writing voice or tone.
03 — Core Concepts

Key Concepts You Need to Know

📖 Key Terminology

Base Model
A model trained only on next-token prediction with no instruction data. Examples: Llama-3-8B (base), Mistral-7B-v0.1.
Continued Pre-Training (CPT)
Running the same pre-training objective on new domain text, starting from a pre-trained checkpoint rather than random weights.
LoRA (Low-Rank Adaptation)
A parameter-efficient method that adds small trainable weight matrices alongside frozen base model weights, drastically reducing memory and compute.
QLoRA
LoRA + 4-bit quantisation of the base model. Reduces VRAM by ~75% making large model fine-tuning feasible on consumer GPUs.
Catastrophic Forgetting
When fine-tuning causes the model to lose previously learned knowledge. LoRA helps mitigate this because base weights remain frozen.
Embed Tokens / LM Head
The input embedding table and output projection layer. For CPT with new tokens (e.g. new language), these must also be trained — not just LoRA layers.
rsLoRA (Rank-Stabilised LoRA)
A LoRA variant that scales alpha by 1/√rank instead of 1/rank, stabilising training at high ranks (≥128) which CPT benefits from.

LoRA Intuition — How It Works

Full fine-tuning updates every parameter in the model (billions of weights). LoRA instead freezes all original weights and adds two tiny matrices A and B alongside each attention layer. Only A and B are trained.

W (frozen)
Base Weight
d × d
Pre-trained, not updated
+
B × A
(d×r) × (r×d)
rank r ≪ d
Only these are trained!

If a weight matrix is 4096×4096 (≈16M params) and LoRA rank r=64, then A is 4096×64 and B is 64×4096 — only ~500K params. That's a 32× reduction in trainable parameters for that layer.

🟢

For CPT specifically: Research (and Unsloth's own blog) shows you should train the embed_tokens and lm_head in addition to LoRA layers. New domain vocabulary (especially new languages or domain-specific tokens) needs the embedding table updated. Use a 2–10× smaller learning rate for these than for the LoRA layers to avoid destabilising training.

Catastrophic Forgetting

One major risk in CPT is that aggressively training on domain data can cause the model to "forget" its general capabilities. Signs include:

  • Model starts generating only in the target language even when prompted in English
  • General reasoning capability degrades
  • Common sense knowledge disappears
⚠️

Mitigation strategies: Use LoRA (base weights stay frozen). Mix a small amount (~5–10%) of general text (e.g. Wikipedia) into your domain corpus. Use a lower learning rate than typical SFT. Monitor eval loss on a held-out general benchmark alongside your domain eval set. Fewer epochs (1–2) is usually safer than 5+.

04 — End-to-End Pipeline

Training Pipeline — Step by Step

01

Collect & Clean Domain Corpus

Gather raw text files from your domain: PDFs converted to text, web-scraped articles, database exports. Clean aggressively — remove HTML tags, headers/footers, duplicate paragraphs, and encoding errors. Quality beats quantity here.

02

Format as HuggingFace Dataset

Structure data as a dataset with a single text column. Each row is a document or a chunk. For CPT there are no instruction/response fields — just raw text.

03

Load Base Model with Unsloth + QLoRA

Use FastLanguageModel.from_pretrained() with load_in_4bit=True. Then call get_peft_model() specifying LoRA rank, alpha, and which layers to target (all linear layers + embed_tokens + lm_head for CPT).

04

Define Training Arguments

Key hyperparameters: max_seq_length (typically 2048), per_device_train_batch_size, gradient_accumulation_steps, learning_rate (2e-4 for LoRA, 2e-5 for embeddings), num_train_epochs, warmup_ratio, lr_scheduler_type.

05

Apply Text Completion Data Collator

Unlike SFT, CPT does NOT mask the prompt tokens. Loss is computed over all tokens. Use DataCollatorForLanguageModeling with mlm=False or Unsloth's built-in text-completion formatting function.

06

Train & Monitor Loss

Launch training with the HuggingFace Trainer API (or TRL's SFTTrainer in CPT mode). Watch training loss — it should decrease steadily. An early spike followed by stagnation often means the learning rate is too high.

07

Save LoRA Adapter & Run Inference

Save the lightweight adapter with model.save_pretrained("my-cpt-adapter"). For inference, load the base model + adapter together. Optionally merge into the base model and push to HuggingFace Hub.

Step 1 in Detail — Data Preparation

The dataset format for CPT is simple — a single text column. Below is how to build one from a list of plain text documents:

python 
from datasets import Dataset

# Example: loading raw text files from a folder
import os

documents = []
for fname in os.listdir("domain_corpus/"):
    with open(f"domain_corpus/{fname}", "r") as f:
        text = f.read().strip()
        if len(text) > 200:   # skip very short docs
            documents.append({"text": text})

# Create a HuggingFace Dataset
dataset = Dataset.from_list(documents)

# Optional: push to Hub
# dataset.push_to_hub("your-username/domain-corpus")

print(dataset)
# Dataset({features: ['text'], num_rows: 15000})
🔍

Data quality checklist: Remove boilerplate repeated across documents (headers, footers, disclaimers). Deduplicate at paragraph level. Filter documents shorter than 200 characters. For PDFs, prefer pdfminer or pypdf2 over OCR if text layer is present. Aim for at least 50MB of cleaned text for meaningful CPT.

Step 3 in Detail — Unsloth Model Setup

python 
# Install Unsloth (run in Colab cell)
# !pip install unsloth --break-system-packages

from unsloth import FastLanguageModel
import torch

# ── Configuration ──────────────────────────────────
max_seq_length = 2048    # max tokens per training sample
dtype = None              # auto-detect (bfloat16 on Ampere+)
load_in_4bit = True      # QLoRA: 4-bit quantised base model

# ── Load Base Model ────────────────────────────────
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/llama-3-8b-bnb-4bit",  # base (not instruct!)
    max_seq_length = max_seq_length,
    dtype = dtype,
    load_in_4bit = load_in_4bit,
)

# ── Apply LoRA ─────────────────────────────────────
model = FastLanguageModel.get_peft_model(
    model,
    r = 128,              # higher rank for CPT (vs 8-64 for SFT)
    target_modules = [   # all linear layers + embeddings
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",
        "embed_tokens", "lm_head",  # ← crucial for CPT
    ],
    lora_alpha = 32,
    lora_dropout = 0,     # 0 is optimal per Unsloth's testing
    bias = "none",
    use_gradient_checkpointing = "unsloth",
    random_state = 42,
    use_rslora = True,    # rank-stabilised LoRA (important at r=128)
)

model.print_trainable_parameters()
# Trainable params: ~85M of 8B (≈1% of total)

Step 4–6 in Detail — Training Configuration

python 
from trl import SFTTrainer
from transformers import TrainingArguments
from unsloth.chat_templates import get_chat_template

# ── Format dataset for text completion (CPT) ───────
# No chat template! Just raw text with EOS token appended.
EOS_TOKEN = tokenizer.eos_token

def format_text(batch):
    return {"text": [doc + EOS_TOKEN for doc in batch["text"]]}

dataset = dataset.map(format_text, batched=True)

# ── Training Arguments ─────────────────────────────
training_args = TrainingArguments(
    output_dir = "./cpt-output",
    num_train_epochs = 1,          # 1-2 epochs for CPT to avoid overfitting
    per_device_train_batch_size = 2,
    gradient_accumulation_steps = 4,  # effective batch size = 8
    warmup_ratio = 0.1,
    learning_rate = 2e-4,
    fp16 = not torch.cuda.is_bf16_supported(),
    bf16 = torch.cuda.is_bf16_supported(),
    logging_steps = 50,
    save_steps = 200,
    save_total_limit = 3,
    optim = "adamw_8bit",           # memory-efficient optimiser
    lr_scheduler_type = "cosine",
    seed = 42,
    report_to = "none",            # change to "wandb" for tracking
)

# ── Trainer (used in CPT/text-completion mode) ─────
trainer = SFTTrainer(
    model = model,
    tokenizer = tokenizer,
    train_dataset = dataset,
    dataset_text_field = "text",    # point to the raw text column
    max_seq_length = max_seq_length,
    dataset_num_proc = 2,
    packing = True,                 # pack multiple docs into one context window
    args = training_args,
)

# ── Train! ─────────────────────────────────────────
trainer.train()
⚠️

packing=True is important for efficiency. It concatenates multiple short documents into a single training sample up to max_seq_length, meaning you don't waste compute on padding. Always use it for CPT unless your documents are already very long.

05 — Complete Example

Complete Minimal CPT Script

A copy-pasteable notebook cell sequence for Google Colab (T4 GPU). This example performs CPT on a small medical text dataset.

python · complete example 
## ═══════════════════════════════════════════════════
## CLASS 21 — Non-Instruction Fine-Tuning
## Continued Pre-Training with Unsloth + QLoRA
## Platform: Google Colab (T4, 16GB VRAM)
## ═══════════════════════════════════════════════════

# CELL 1: Install
# !pip install unsloth datasets trl -q --break-system-packages

# CELL 2: Imports
from unsloth import FastLanguageModel
from datasets import load_dataset, Dataset
from trl import SFTTrainer
from transformers import TrainingArguments
import torch

# CELL 3: Config
MODEL_NAME = "unsloth/llama-3-8b-bnb-4bit"
MAX_SEQ_LEN = 2048
LORA_RANK   = 128
EPOCHS      = 1
LR          = 2e-4
BATCH_SIZE  = 2
GRAD_ACC    = 4

# CELL 4: Load model
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name=MODEL_NAME, max_seq_length=MAX_SEQ_LEN,
    load_in_4bit=True,
)
model = FastLanguageModel.get_peft_model(
    model, r=LORA_RANK, use_rslora=True,
    target_modules=[
        "q_proj","k_proj","v_proj","o_proj",
        "gate_proj","up_proj","down_proj",
        "embed_tokens","lm_head",
    ],
    lora_alpha=32, lora_dropout=0, bias="none",
    use_gradient_checkpointing="unsloth", random_state=42,
)

# CELL 5: Load dataset (replace with your domain corpus)
# Using HuggingFace's medical_meadow_pubmed_causal as example
raw = load_dataset("medalpaca/medical_meadow_pubmed_causal",
                   split="train[:5000]")
# Combine input+output into one raw text blob
def merge_text(batch):
    texts = [i + " " + o + tokenizer.eos_token
             for i, o in zip(batch["input"], batch["output"])]
    return {"text": texts}
dataset = raw.map(merge_text, batched=True,
                   remove_columns=raw.column_names)

# CELL 6: Train
trainer = SFTTrainer(
    model=model, tokenizer=tokenizer,
    train_dataset=dataset, dataset_text_field="text",
    max_seq_length=MAX_SEQ_LEN, packing=True,
    args=TrainingArguments(
        output_dir="./cpt_out", num_train_epochs=EPOCHS,
        per_device_train_batch_size=BATCH_SIZE,
        gradient_accumulation_steps=GRAD_ACC,
        learning_rate=LR, warmup_ratio=0.1,
        lr_scheduler_type="cosine", optim="adamw_8bit",
        bf16=torch.cuda.is_bf16_supported(),
        fp16=not torch.cuda.is_bf16_supported(),
        logging_steps=25, seed=42, report_to="none",
    ),
)
trainer_stats = trainer.train()
print(f"Training time: {trainer_stats.metrics['train_runtime']:.1f}s")

# CELL 7: Inference
FastLanguageModel.for_inference(model)
prompt = "Acute myocardial infarction is caused by"
inputs = tokenizer([prompt], return_tensors="pt").to("cuda")
out = model.generate(**inputs, max_new_tokens=150,
                       temperature=0.7, do_sample=True)
print(tokenizer.decode(out[0], skip_special_tokens=True))

# CELL 8: Save
model.save_pretrained("my-medical-cpt-adapter")
tokenizer.save_pretrained("my-medical-cpt-adapter")
06 — Practitioner Notes

Tips, Pitfalls & Best Practices

✅ Things that work well

  • ✓ Use a base model (not an instruct model) as your starting point for CPT — you'll apply instruction tuning in the next step anyway.
  • ✓ Enable packing — it speeds up training significantly by eliminating wasted padding tokens.
  • ✓ Include embed_tokens and lm_head in your LoRA targets, especially if your domain has unusual vocabulary.
  • ✓ Use rsLoRA (use_rslora=True) when rank ≥ 128 — standard LoRA is unstable at high ranks.
  • ✓ Mix general text (5–10%) into your domain corpus to reduce catastrophic forgetting.
  • ✓ Start with 1 epoch. Check if domain perplexity improved before deciding to run more.

❌ Common mistakes

  • ✗ Fine-tuning on an instruction model when you want CPT — you risk overwriting the instruction-following behaviour.
  • ✗ Using a chat template for CPT — it adds special tokens that aren't appropriate for raw text completion training.
  • ✗ Setting a learning rate that's too high (e.g. 1e-3) — causes divergence, especially with embed_tokens/lm_head included.
  • ✗ Training for too many epochs on a small dataset — the model memorises training text and loses generalisation.
  • ✗ Skipping data cleaning — noisy data (OCR errors, HTML tags, duplicates) degrades model quality significantly.

📐 Hyperparameter Reference

Hyperparameter CPT Recommendation Why
LoRA rank (r) 64–256 CPT needs higher capacity than SFT to absorb new knowledge
Learning rate 1e-4 to 2e-4 Lower than SFT to avoid catastrophic forgetting
Embed LR multiplier 0.1× to 0.5× of LoRA LR Embedding tables are sensitive; smaller updates stabilise training
Epochs 1–2 More epochs risks overfitting and forgetting
Context length 2048–4096 Longer is better for coherent document-level learning
Batch size (effective) 8–32 Larger batches stabilise gradient updates on heterogeneous text
07 — Self-Check

Quiz — Test Your Understanding

📝 Concept Check Questions

1. What is the key difference between Continued Pre-Training and Supervised Fine-Tuning in terms of data format?

▶ Show Answer
CPT uses raw, unlabelled text (no instruction/response pairs). The model simply predicts the next token on all text. SFT uses prompt–response pairs and typically masks the loss on the prompt so only the response tokens contribute to the gradient.

2. Why should you include embed_tokens and lm_head in your LoRA targets for CPT but not necessarily for SFT?

▶ Show Answer
CPT often introduces new domain vocabulary or a new language that the base model's embedding table has not seen. Without updating embed_tokens and lm_head, the model cannot properly represent or generate domain-specific tokens. For SFT (instruction following), the vocabulary is usually unchanged, so updating these is less critical.

3. What is catastrophic forgetting and what is the primary mechanism LoRA uses to mitigate it?

▶ Show Answer
Catastrophic forgetting is when a neural network loses previously learned knowledge while being trained on new data. LoRA mitigates this by keeping all original (pre-trained) weight matrices completely frozen. Only the small A and B adapter matrices are updated, so the base model's general knowledge is preserved.

4. What does packing=True do in the SFTTrainer, and why is it especially useful for CPT?

▶ Show Answer
packing=True concatenates multiple training documents into a single sequence up to max_seq_length, filling the context window efficiently. It's valuable for CPT because domain text documents are often much shorter than the context window, so without packing most of each training batch would be padding — wasting compute and slowing training.

5. Should you start CPT from a base model or an instruction-tuned model? Why?

▶ Show Answer
Generally, start from a base model. The standard pipeline is: (1) base model → CPT → domain-adapted base model → (2) SFT → domain instruction model → (3) DPO → aligned model. If you CPT an already instruction-tuned model, you risk degrading the instruction-following capability you'd want to preserve, and you'd typically follow up with SFT anyway to re-instil it.

6. What is rsLoRA and when should you use it?

▶ Show Answer
rsLoRA (Rank-Stabilised LoRA) scales the LoRA output by 1/√rank instead of 1/rank. At higher ranks (≥64), standard LoRA can become numerically unstable because the effective scale of the adapter grows with rank. rsLoRA compensates for this, making high-rank LoRA training stable — which is recommended for CPT since CPT benefits from higher ranks than SFT.
